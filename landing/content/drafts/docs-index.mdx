PayKit is a TypeScript-first, server-side payments orchestration framework for modern SaaS.
It sits between your app and payment providers, normalizes billing events, and keeps the
minimum billing state your app needs locally.

PayKit does **not** process payments. Providers still run checkout, vault payment methods, and
move money. PayKit owns the orchestration layer above them.

<Callout type="warn">
  These docs describe the planned MVP surface. The package is not public yet, so code examples are
  mock examples written against the intended server API.
</Callout>

## What the MVP covers

The first docs story follows one narrow loop:

1. Create a provider-hosted checkout or payment method flow
2. Receive provider webhooks
3. Normalize those webhooks into one PayKit event model
4. Sync billing state into PayKit-owned Postgres tables
5. Charge a saved payment method later through one server API

Subscriptions are intentionally **not** the core docs story right now.

## Architecture

```txt
Your App -> PayKit Core -> Provider Adapter -> External Provider
```

That split matters:

- your app calls one server-side API
- providers stay behind thin adapters
- webhook payloads become provider-agnostic PayKit events
- your app reads normalized billing state instead of stitching together provider APIs

## Core principles

- **Orchestration, not processing** — PayKit coordinates providers, it does not move money
- **Local state is part of the product** — PayKit persists the billing state your app needs
- **Provider-agnostic** — business code should not depend on provider-native IDs or event names
- **Postgres first** — the MVP assumes PayKit-owned `paykit_*` tables
- **Keep the abstraction narrow** — the initial promise is checkout, saved methods, webhooks, and charges
